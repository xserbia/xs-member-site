3 estados
Inicial initial
Sin datos
No existe financialRadiografiaResults ni financialRadiografiaInProgress en localStorage. Muestra CTA "Hacer radiografía".
Incompleto incomplete
En progreso
Existe financialRadiografiaInProgress pero no financialRadiografiaResults. Muestra porcentaje calculado como round(currentQ / 14 × 100) y CTA "Continuar radiografía".
Completado completed
Con resultado
Existe financialRadiografiaResults. Muestra etapa, barra de progreso combinada y checklist de áreas a mejorar. CTA "Actualizar radiografía".

3. Lógica de etapas

Etapa 1 Supervivencia

Estás luchando para cubrir gastos básicos o no tienes ningún colchón.

Condición de entrada
A) cashFlow === "no" + emergency < 1
A) cashFlow === "ahorros" + emergency < 1
A) cashFlow === "justo" + emergency < 1

Si cashFlow es cualquiera de los tres primeros Y emergency < 1 → Etapa 1. Si emergency >= 1 → Etapa 2.

Condiciones de salida (ambas)
✓ cashFlow = "justo" o "sobra"
✓ emergency >= 1 mes de gastos esenciales
No se pide en esta etapa
Ahorro porcentual · Seguros · Reducir deudas · Invertir
Etapa 2 Seguridad

Flujo positivo pero sin protección completa ante imprevistos.

Condición de entrada (cualquiera)
emergency < idealEmergency
debt >= 36% ingreso neto (o >= 30% bruto)
debtRatio >= 50%
insuranceScore < 100
estateScore < 100
Condiciones de salida (todas)
✓ emergency >= idealEmergency
✓ debt < 36% neto / < 30% bruto
✓ debtRatio < 50%
✓ insuranceScore = 100
✓ estateScore = 100 (básico)
Etapa 3 Acumulación

Construyendo riqueza consistentemente. Meta: alcanzar Stanley Great para la edad.

Condición de entrada (cualquiera)
netWorth < stanleyGreat para su edad
save < 20% del ingreso
debt >= 20% del ingreso
debtRatio >= 30%
passive < 25% de gastos
Condiciones de salida (todas)
✓ netWorth >= stanleyGreat
✓ save >= 20%
✓ debt < 20%
✓ debtRatio < 30%
✓ passive >= 25% de gastos
Fuentes de ingreso pasivo válidas
Títulos financieros Dividendos / ETFs Bienes raíces / REITs Negocios automatizados Regalías / licencias
Etapa 4 Independencia

Las inversiones pueden mantener el estilo de vida. Ingresos pasivos cubriendo gastos.

Condición de entrada (cualquiera)
passive < 100% de gastos
debt >= 10% del ingreso
estateScore < 100 (completo)
Condiciones de salida (todas)
✓ passive >= 100% de gastos
✓ debt < 10% del ingreso
✓ estateScore = 100 (incl. trust)
Etapa 5 Libertad

Libertad financiera completa. Sin issues pendientes en ninguna dimensión.

Metas permanentes (no son condiciones de salida)
Preservar y proteger el patrimonio acumulado
Optimización fiscal avanzada
Planificación de legado generacional
Impacto filantrópico

Metas Financieras — Modelo de Datos

metas[]
Estructura por ítem: id name amount · el índice en el array define la prioridad (0 = prioridad #1)
idstring
Math.random().toString(36).slice(2, 10) UID generado en cliente al crear la meta. 8 caracteres alfanuméricos. No se reutiliza.
namestring
texto libre — máx. sin límite explícito Nombre de la meta. Se valida que no sea vacío y que no exista duplicado (comparación case-insensitive). Origen: input manual o selección del catálogo.
amountnumber
float > 0 Monto objetivo en la moneda del cliente. Se valida que sea > 0. Se muestra formateado con símbolo y locale de la moneda seleccionada.
prioritynumber — derivado
índice + 1 (posición en el array)
derivado
No se guarda en el ítem. Se deriva del orden del array. Al reordenar (drag & drop), se hace splice + insert en el array y se re-renderiza. Al exportar a discovery_draft_v4.goals se incluye como campo explícito.

LocalStorage

2 claves
metas_financieras_v2 clave propia
JSON → [ { id, name, amount }, … ]
Array ordenado
Array de metas en el orden de prioridad del usuario. Se escribe en cada operación (agregar, eliminar, reordenar). El orden del array ES la prioridad — el índice 0 es prioridad #1.
discovery_draft_v4 clave compartida
JSON → { …draft, goals: [ { name, amount, priority }, … ] }
Sincronización
Solo se sincroniza si el draft ya existe y tiene contenido. Las metas se inyectan en el campo goals con priority explícito (índice + 1). También lee discovery_draft_v4.currency para determinar símbolo y locale de formato.

Comportamiento y Validaciones

reglas de negocio
Agregar meta
Permite Bloquea
Botón deshabilitado hasta que name.length > 0 AND amount > 0. Bloquea duplicados (case-insensitive) con toast de aviso. Al agregar: push al final del array, limpiar inputs, re-enfocar name input.
Reordenar
splice(fromIdx, 1) → splice(toIdx, 0, moved)
Mouse drag & drop Touch (long press 150ms)
Delegado en el contenedor de la lista. Soporta tanto mouse (HTML draggable) como touch (touchstart/touchmove/touchend). Los ítems reciben draggable="true" dinámicamente vía MutationObserver después de cada render.
Eliminar meta
metas.filter(m => m.id !== id) Sin confirmación. Inmediato. Se actualiza el array, se re-renderiza y se muestra toast. La prioridad de todos los ítems siguientes sube automáticamente (re-indexación implícita).
Etiquetas de rango
RANK_LABELS = ["#1 Prioridad", "#2", "#3", "#4", "#5", "#6", "#7", "#8"] Las primeras 3 posiciones tienen estilos visuales diferenciados (dorado, plateado, bronce). A partir del índice 8 se usa "#" + (idx + 1) como fallback.
Total acumulado
metas.reduce((s, m) => s + m.amount, 0) Suma de todos los montos objetivo. Se muestra en una card al pie cuando hay al menos 1 meta. Se oculta cuando la lista está vacía.

Coaching Process — Pipeline de Pasos

3 pasos lineales
Array steps[]: id name icon order
Paso 1 id: "discovery"
Discovery El cliente completa el formulario externo. El progreso se refleja en el dashboard automáticamente.
Paso 2 id: "videocall"
Presentación Video personalizado pregrabado. Tiene dos subestados: en preparación (videoReady=false) y disponible (videoReady=true). WhatsApp como canal de soporte.
Paso 3 id: "cerrado"
Cerrado Entrega del documento final. Contiene el Money Map Summary modal y soporte WhatsApp.
El pipeline es estrictamente lineal y secuencial. Un paso solo se desbloquea cuando todos los ítems del checklist del paso anterior están marcados como done: true. No hay saltos ni retrocesos forzados.

Estructura de Datos por Paso

stepContent{}
Campos comunes a todos los pasos: title description checklist[] cta
titlestring
Título del paso que se muestra en el card de contenido derecho.
descriptionstring
Descripción larga del paso. Explica al cliente qué hacer y qué esperar.
checklist[]{ item: string, done: boolean }[]
[ { item: "texto del ítem", done: false }, … ] Array de ítems de verificación. Cuando todos son done: true, se dispara el auto-advance al siguiente paso (con 600ms de delay). Los ítems se marcan programáticamente desde las funciones de cada paso — el usuario no los toca directamente.
cta{ text, action } | null
Botón CTA opcional para pasos que no tienen sección dinámica propia. Los pasos discovery, videocall y cerrado usan cta: null porque tienen su propio HTML builder.

Estados y Transiciones

currentStep / selectedStep
currentStepstring — paso activo
"discovery" | "videocall" | "cerrado" Determina hasta qué paso están desbloqueados. Un paso es accesible si su índice ≤ índice de currentStep. Avanza automáticamente cuando el checklist del paso actual se completa.
selectedStepstring — paso en vista
"discovery" | "videocall" | "cerrado" El paso que el usuario está viendo actualmente. Puede ser diferente de currentStep si el usuario navega a un paso anterior. Las notas se cargan según selectedStep.
getStepStatus(step)función
stepIdx < currentIdx → "completed"
stepIdx === currentIdx → "current"
stepIdx > currentIdx → "pending" Determina el ícono y etiqueta de cada paso en el timeline. "completed" = check-circle verde, "current" = ícono del paso, "pending" = lock gris.
Auto-advancelógica de transición
if (allDone && nextStep && currentStep === selectedStep)
→ setTimeout(600ms) → currentStep = nextStep.id
Solo cuando selectedStep === currentStep
No avanza si el usuario está viendo un paso anterior. Al avanzar: actualiza currentStep, selectedStep, re-renderiza timeline, actualiza progreso y carga notas del nuevo paso.
Barra de progresoheader sidebar
completedCount / steps.length × 100 = % Cuenta los pasos con status "completed". Se muestra como barra + texto "X/3".

Paso 1 — Discovery

discoveryStatus{}
Campos de discoveryStatus: status progress lastUpdated link summary{}
statusenum
not_started in_progress completed
Controla la UI del panel Discovery: estado not_started muestra alerta + botón "Iniciar". in_progress muestra barra de progreso + botón "Continuar". completed muestra resumen de respuestas.
progressnumber — 0 a 100
Porcentaje de completado del formulario externo. Se actualiza programáticamente cuando el formulario externo manda señales (o manualmente en demo). Cuando llega a 100 → status = "completed".
linkstring — URL
https://xserbia.github.io/xavierserbia-site/discovery-form-claude.html Se abre en nueva pestaña. Al abrir, si status === "not_started", después de 2s cambia a in_progress con progress=10 y marca checklist[0].done=true.
summaryobject
{ businessName, industry, currentRevenue, goals } Resumen de respuestas clave del formulario. Solo se muestra cuando status === "completed". Se popula programáticamente al completar el Discovery.
Checklist items3 ítems
[0] "Formulario iniciado" → done cuando status cambia a in_progress [1] "Información básica completada" → done cuando progress ≥ 70 [2] "Discovery enviado para revisión" → done cuando status === "completed"

Paso 2 — Presentación

videocall{}
Campos específicos: videoReady videoUrl whatsappNumber whatsappMessage
videoReadyboolean
false → "en preparación" true → "video disponible"
Controla cuál de los dos estados se renderiza. false: muestra card de "presentación en preparación" con timeline de 3 pasos (discovery ✓, grabando ●, lista ○). true: muestra card de WhatsApp support y el video es reproducible.
videoUrlstring
URL del video personalizado. Si está vacío, al hacer clic en play se muestra notificación "En producción aquí se reproducirá el video". En producción apunta al video real del cliente.
whatsappNumberstring
formato: "15551234567" (sin + ni espacios) Número de WhatsApp del asesor. Se usa en la URL https://wa.me/{number}?text={msg} en los pasos Presentación y Cerrado.
whatsappMessagestring
Mensaje pre-rellenado al abrir WhatsApp. Se encodea con encodeURIComponent(). Paso Cerrado tiene su propio mensaje hardcodeado.
Checklist items3 ítems
[0] "Presentación lista" → done cuando videoReady=true o al reproducir [1] "Video visto" → done al reproducir o al marcar "visto" [2] "Preguntas resueltas" → done al abrir WhatsApp

Paso 3 — Cerrado

cerrado{}
Campos específicos: finalDocuments[]
finalDocuments[]array
{ name, description, size, pages, type, status, icon, url }
status: "pending" → bloqueado status: "available" → descargable
Array de documentos finales entregables. Cuando status === "available" se muestra el botón de descarga y el mensaje de celebración. Cuando status === "pending" se muestra el estado "en preparación" con alerta informativa.
Campos del documento
name — nombre del documento mostrado en la card description — descripción del contenido size — tamaño del archivo (ej. "2.4 MB") pages — número de páginas type — tipo de archivo ("pdf") icon — nombre del ícono Lucide (ej. "file-text") url — URL de descarga del documento
Checklist items1 ítem
[0] "Resumen final entregado" → done cuando finalDocuments[0].status === "available"
Secciones del panel
1. Money Map Summary banner (siempre visible) → abre modal mmsOverlay 2. Header del estado del documento (available vs pending) 3. Card del documento final con botón descarga 4. Mensaje de felicitaciones (solo cuando available) 5. Card de soporte WhatsApp (siempre visible)

LocalStorage

1 clave
userDashboardNotes notas por paso
JSON → { "discovery": "texto…", "videocall": "texto…", "cerrado": "texto…" }
Por paso
Mapa de stepId → string. Se carga al seleccionar un paso y se guarda al pulsar el botón "Guardar" de la sección de notas. Las notas son independientes por paso.
El resto del estado del proceso (stepContent, currentStep, progreso del discovery) NO se persiste en localStorage — vive solo en memoria durante la sesión. El servidor es la fuente de verdad para el estado del proceso.
